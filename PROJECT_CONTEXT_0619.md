## 6. 후속 논의 — CSV 인코딩 깨짐 문제 & 엑셀 업로드 전환 검토 (2026-06-18, 외부 인터넷망에서 작성)

> 이 섹션은 **외부 인터넷망 환경**에서 작성되었습니다. 다음 날 **은행망**에서 이어받는 Claude(또는 작업자)는 아래 내용을 먼저 읽고, 사용자가 가져올 "망간전송 .xlsx 허용 여부" 확인 결과에 따라 다음 단계를 진행하세요.

### 6.1 문제 상황
사용자가 실측 데이터를 CSV 템플릿에 채워 넣는 실제 워크플로우는 다음 5단계입니다.

1. (현재 환경) 은행 인터넷망 — 여기서 CSV 템플릿(`template_*.csv`)을 생성/보유
2. 망간시스템으로 해당 CSV를 **은행 내부망**(DB 접근 가능한 망)으로 반입
3. 내부망에서 DB 데이터를 정리해 **엑셀로 CSV를 열고** 실측값을 입력
4. 완성된 CSV를 망간전송으로 다시 **은행 인터넷망**으로 반출
5. 인터넷망에서 엑셀로 열어보면 **한글이 한문·외국어 섞인 형태로 와장창 깨짐**

### 6.2 진단 결과
- 깨짐은 망간전송 자체보다 **3번 단계(엑셀에서 CSV를 열고 저장하는 순간)에서 발생할 가능성이 높음**으로 결론.
- 원인: 현재 CSV 템플릿이 **BOM 없는 UTF-8**임. 한국어 윈도우 엑셀에서 BOM 없는 UTF-8 CSV를 더블클릭으로 열면 시스템 로케일(CP949/EUC-KR)로 잘못 추정 → 화면에 즉시 깨진 글자 표시 → 그 상태로 저장하면 ANSI(CP949)로 재인코딩되며 **깨진 내용이 바이트 단위로 영구 박제**됨. 이후 단계(4, 5)는 이미 손상된 파일을 옮기고 다시 여는 것뿐.
- 검증: 실제로 `template_1_weekly_ev331.csv`의 바이트를 확인해 BOM이 없음을 확인함 (`77 65 65 6B` = "week..."로 시작, `EF BB BF` 없음).
- 테스트용으로 BOM을 추가한 `template_1_weekly_ev331_BOM테스트.csv` 파일을 작업 폴더에 생성해둠 (검증/실험용, 실제 템플릿 대체 아님 — 필요 없으면 삭제 가능).
- `standalone.html`의 CSV 파서(`parseRows`, [standalone.html:303-328](standalone.html#L303-L328))는 `FileReader.readAsText()`를 쓰는데, 표준상 BOM을 자동으로 제거해주므로 **BOM이 있는 CSV를 업로드해도 대시보드 자체는 문제없음**을 코드로 확인함.

### 6.3 검토한 해결 방향 (CSV 유지 시)
- 엑셀에서 CSV를 **더블클릭으로 열지 말고**, "데이터 ⟩ 텍스트/CSV에서 가져오기" 마법사로 "파일 원본"을 직접 "유니코드(UTF-8)"로 지정해서 가져오기 (가장 신뢰도 높은 방법, 엑셀 버전 대부분에 있음)
- 저장 시 일반 "CSV (쉼표로 분리)"가 아니라 **"CSV UTF-8(쉼표로 분리)"** 형식 선택 (없으면 메모장으로 UTF-8 BOM 재저장 우회)
- 템플릿 자체를 미리 UTF-8 BOM으로 배포 (1차 방어선) — 이 작업은 아직 실행 안 함, 사용자 확인 대기 중

### 6.4 사용자가 제안한 더 근본적인 방향 — CSV 대신 엑셀(.xlsx) 업로드로 전환
- 논리: .xlsx는 zip+XML 바이너리라 "인코딩 추측" 단계가 구조적으로 없음 → 위 3중 추측 구간(엑셀 열 때/저장할 때/재오픈할 때)이 전부 사라짐. 기술적으로는 가장 깨끗한 해법.
- **블로커 — 다음 날 반드시 확인 필요**: 은행 망간전송 시스템이 `.xlsx` 확장자 반출/반입을 허용하는지. (CSV보다 느슨할 수도, 매크로/OLE 위험 때문에 더 엄격하거나 별도 승인이 필요할 수도 있음 — 이건 사용자가 사내 보안팀/IT에 확인해야 알 수 있는 부분이라 미확정 상태로 둠.)
- 허용된다면 진행할 구현 범위 (아직 코드 작업 시작 안 함):
  1. SheetJS(xlsx) 라이브러리를 esm.sh CDN으로 추가 (현재 React/Recharts/lucide와 동일한 importmap 방식)
  2. `FileReader.readAsText` → `readAsArrayBuffer`로 변경, 기존 파서 4종(`parseSimBase`/`parseWeekly`/`parseMonthly`/`parseFunnelWide`)의 입력단만 xlsx 시트→행 변환으로 교체 (출력 row 객체 구조는 그대로 유지 → 이후 계산 로직은 무수정)
  3. 9개 CSV 템플릿을 동일 헤더 구조의 xlsx 템플릿으로 재생성
  4. 업로드 input의 `accept=".csv"` → `accept=".xlsx"` 변경 (4곳: 기준값 CSV, 주간 실적 CSV, 퍼널 CSV, 고신용/중저신용 월간 수익성 CSV)
  5. **주의**: `standalone.html`은 git 미관리 단일 파일이므로 ([본 문서 §1](#1-이-프로젝트는-무엇인가)) 작업 전 백업 필수

### 6.5 다음 세션(은행망)에서 할 일
1. 사용자에게 "망간전송 .xlsx 허용 여부" 확인 결과를 물어볼 것

## 7. 후속 논의 — `elasticity`(금리 탄력성 계수) 추정 방법론 검토 (2026-06-18, 외부 인터넷망에서 작성)

> 이 섹션도 외부 인터넷망에서 작성됨. 6번 항목(인코딩/xlsx)과는 별개의 독립된 주제 — `template_2_simbase.csv`의 `elasticity` 컬럼(현재 가정값)을 실측치로 교체하는 작업의 방법론 논의.

### 7.1 배경
- `elasticity`(ε)는 "금리 10bp 변화 시 이탈률(churn)이 변하는 정도"를 나타내는 계수. 공식: `churnNew = churn + (bp/10) × elasticity` ([standalone.html](standalone.html) `computeRows`, [CSV_데이터_가이드.md:53](CSV_데이터_가이드.md#L53)).
- 현재는 등급별 **가정값**으로 채워져 있고, 요구사항정의서 v0.3 §7 Open Item #2에 "ADW 실측 데이터 확보 후 역산하여 교체 필요"로 명시되어 있던 항목.
- CSV가이드에 이미 기본 역산 공식이 적혀 있음: "과거 금리 변경 이벤트 전후 이탈률 변화량 ÷ (bp변화/10)" ([CSV_데이터_가이드.md:79](CSV_데이터_가이드.md#L79)). 오늘 논의는 이 기본 공식을 그대로 쓰면 표본(3개 이벤트)이 너무 적어 노이즈가 크다는 문제의식에서 시작해, 더 안정적인 추정 방법을 설계해본 것.

### 7.2 데이터 가용성 확인
- **bp 변동값**: 사용자가 과거 3건 이벤트(3/31, 4/23, 5/29)의 등급별 실제 bp 조정값을 충분히 제공 가능하다고 확인함. (단, "등급별로 bp가 실제로 달랐는지"는 아직 미확인 — 7.5 참고)
- **이탈률(churn)**: `template_4_funnel_ev*.csv`의 `preApprovalExec_before/after`로부터 `churn = 100 − preApprovalExec`로 등급별·이벤트별 도출 가능 (이미 CSV가이드에 개념적 연결관계가 명시되어 있던 부분).
- 즉 (Δbp, Δchurn) 쌍을 이벤트 3개 × 등급 7개 = 최대 21개까지 만들어낼 수 있는 셋업.

### 7.3 사용자가 제기한 우려 — 두 가지 교란요인
1. **경쟁사 금리조정 추이**: 추적 불가능 (외부사 데이터 접근 불가)
2. **기준금리(시장금리)**: 매일 변동하는데 추적/정렬이 번거로움. 다만 사용자 직관: "기준금리는 모든 은행에 거의 유사하게 작용하니 굳이 고려 안 해도 될 것 같다"

### 7.4 제안한 해법 — 이벤트(시점) 고정효과 + 등급간 변동으로 식별
핵심 통찰: 경쟁사 움직임이든 기준금리든, **"그 시점에 모든 등급에 공통으로 걸리는 충격"**이라는 점에서 동일함. 따라서 같은 이벤트 내에서 등급별로 bp 조정폭이 다르게 들어갔다면, 등급 간 차이를 보는 순간 공통충격이 자동으로 상쇄(차분)됨 — 경쟁사/기준금리 데이터를 직접 모으지 않아도 통계적으로 통제되는 효과.

```
모형: Δchurn_(등급, 이벤트) = α_이벤트(이벤트별 고정효과 — 공통충격 흡수)
                            + ε × (Δbp_(등급, 이벤트) / 10) + 잔차
```

- 사용자의 "기준금리는 비슷하게 움직이니 제외해도 될 듯"이라는 직관을 경쟁사 효과까지 포함해 통계적으로 정당화하는 설계.
- **성립 조건**: 이벤트 내에서 등급별 bp 조정폭이 실제로 달라야 함. 매 이벤트마다 전 등급에 동일한 bp를 일괄 적용했다면 등급 간 비교할 변동 자체가 없어 이 방법이 작동하지 않음.

### 7.5 차선책 — 등급별 bp가 일괄 동일했을 경우
- 이벤트 간(3개 점) 비교만 남게 되므로 공통충격을 통계적으로 못 걷어냄.
- 이 경우 점추정 대신 **민감도 범위**(예: "경쟁사 영향 0~X% 가정 시 elasticity 범위") 형태로 제시하는 것이 차선.
- 기준금리는 경쟁사와 달리 **공개 데이터**(CD 91일물, 금융채 등 — 한국은행 ECOS, 채권정보센터 등)이므로 실제로 가져와 통제변수로 넣는 것까지는 가능. 경쟁사 효과만 끝내 못 잡으므로 그 부분은 잔차 불확실성으로 명시.

### 7.6 다음 세션에서 할 일
1. **사용자가 내일 사내에서 확인할 것**: 과거 3건 이벤트에서 등급별 bp 조정폭이 실제로 달랐는지 (일괄조정이었는지 등급 차등이었는지)
2. **등급별로 달랐다면** → 7.4의 고정효과 회귀로 진행. 필요한 입력: 이벤트별 등급별 실제 적용 bp, funnel CSV의 등급별 churn(전/후)
3. **일괄조정이었다면** → 7.5의 민감도 범위 방식으로 전환, 기준금리(공개 시장금리) 데이터 소싱 방법부터 논의 필요
4. 아직 코드/스프레드시트 작업은 시작 안 함 — 방법론 설계 단계까지만 논의됨
