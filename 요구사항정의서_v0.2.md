## 1. 시스템 개요

### 1.1 목적
사장님신용대출의 **전략등급별 금리(마진율)를 조정했을 때** 발생하는 다음 효과를 한 화면에서 시뮬레이션하고,
실제 금리 변경 시행 후의 **실적 변화를 추적**하기 위한 대시보드.

1. 신규유입 변화(즉시효과): 신규취급액, 순증액, 대출잔액, 대출건수, 신규고객 증가분
2. 손익 변화(균형추정 — 월간 기준): 이자수익, 공헌이익, 영업이익
3. 간접효과: 카드발급고객, 연동 예금평잔, MAU 고객 증가
4. 시행 실적: 과거 3건의 금리 변경 이벤트(3/31, 4/23, 5/29)에 대한 전/후 비교 (주간 영업지표, 월간 수익성, 퍼널 전환율)

### 1.2 기술 스택 / 구현 방식
- **단일 HTML 파일**로 구현 (배포·공유 용이성 우선, 별도 빌드 과정 없음)
- `importmap` + esm.sh CDN을 통해 React 18.3.1 / ReactDOM 18.3.1 / Recharts 2.12.7 / lucide-react 0.383.0 로드
- `<script type="text/babel" data-presets="react" data-type="module">` — Babel Standalone으로 브라우저 내 JSX 인라인 트랜스파일 (오프라인 환경에서는 CDN 접근 필요)
- 차트: Recharts (`BarChart`, `Bar`, `LabelList`, `ResponsiveContainer`, `Tooltip`, `Legend`, `ReferenceLine`, `CartesianGrid`, `XAxis`, `YAxis`)
- 아이콘: lucide-react (`Sliders`, `BarChart3`, `Gauge`, `Wand2`, `Upload`, `AlertTriangle`, `Info`, `TrendingUp`, `TrendingDown`, `Users`, `PiggyBank`, `Activity` 등)
- 상태관리: React `useState` / `useMemo` (전역 상태 라이브러리 없음, `App` 컴포넌트에 전체 상태 보유 후 props로 하위 전달)
- 데이터 입력: 모든 외부 데이터는 **CSV 파일 업로드**를 통해 클라이언트 사이드에서 파싱(FileReader) — 서버/백엔드 없음

### 1.3 전체 화면 구성

## 3. 화면별 요구사항

## 3.1 탭 1 — 금리 조정 시뮬레이터 (`Simulator`)

### 3.1.1 목적
전략등급(1~7등급)별로 금리(마진율)를 ±50bp 범위에서 조정했을 때, 신규유입·이탈률·잔액·수익성에 미치는 영향을
"현재" 대비 "조정 후" 값으로 즉시 비교한다.

### 3.1.2 화면 구성 (상→하 순서)

#### (1) 헤더 영역 (`S.simHeadRow`)
- 좌측: Gauge 아이콘 + "전략등급 금리 조정 시뮬레이터" 제목
- 우측 액션 버튼 3종:
  - **최적금리 찾기(참고)** (Wand2 아이콘) — `findOptimalAdj` 실행, 등급별 공헌이익을 최대화하는 bp 조정값을 자동 산출하여 슬라이더에 반영
  - **초기화** — 모든 등급의 조정값(bp)을 0으로 리셋
  - **기준값 CSV** (Upload 아이콘) — `template_2_simbase.csv` 형식 파일 업로드 → 시뮬레이션 기준값(`simBase`) 전체 교체 + 조정값 초기화

#### (2) 가정값 안내 박스 (`S.assumptionBox`)
- AlertTriangle 아이콘
- 문구: "탄력성 계수 및 원가율 6종은 등급별 **가정값**이며 실측치 확보 시 기준값 CSV로 교체하세요."
- 핵심 산식 노출:
  - **공헌이익 = 이자수익 − 자금원가율 − 법적비용 − 계약신용원가**
  - **영업이익 = 공헌이익 − 계약업무원가 − 신용원가gap − 업무원가gap + 비이자이익**
- 기준값 CSV 헤더 스펙 텍스트: `grade,balance,count,newWk,netWk,churn,margin,legalCostRate,contractCreditRate,contractOpRate,creditGapRate,opGapRate,nonIntRate,elasticity`
- 현재 적용 중인 기준값 파일명 표시 (`현재: {baseFile}` — 기본값 "기본 가정값")
- **자금원가율(cof) 입력 필드**: "(사장님 신용대출 3개월 평균) 자금원가율 [입력칸]%" — 사용자가 직접 입력 (기본값 1.7%), 모든 공헌이익/영업이익 계산에 즉시 반영

#### (3) 등급별 조정 슬라이더 (`S.sliderWrap`)
- 1등급~7등급, 각 행마다:
  - 등급명 표시
  - 슬라이더: `min=-50, max=50, step=5` (bp 단위)
  - 현재 조정값 표시: `{+/-}{bp}bp` (양수=`primary` 색, 음수=`secondary` 색, 0=`faint` 색)
  - 탄력성 계수 표시: `ε {elasticity}`

#### (4) "신규유입 변화" 섹션 — 즉시효과 (`S.subSectionLabel`)
- 부제: "(즉시효과 — 금리 조정 시 단기 반응)"
- 카드 5종 (`volTotals`, KpiCard로 렌더):

| 라벨 | 단위 | 산출 |
|---|---|---|
| 신규취급액 (주) | 억 | `sim.tot.newWkOld → newWkNew` (소수점 1자리, n.n억) |
| 순증액 (주) | 억 | `sim.tot.netWkOld → netWkNew` (소수점 1자리, n.n억) |
| 대출 잔액 | 억 | `sim.tot.balOld → balNew` (소수점 1자리, n.n억) |
| 대출 건수 (주간) | 건 | `sim.tot.cntOld → cntNew` (정수, 천단위 콤마) |
| 신규고객 증가분 | 명 | `sim.indirect.custGain` — 기준값 없이 증감분만 표시(`noBase: true`), 항상 "조정 후 유입 고객 증가분"이라는 고정 설명 |

> **v0.2 변경**: 신규취급액/순증액/대출잔액 3종은 기존 정수(소수점 0자리) 표시에서 **소수점 1자리(n.n억)** 표시로 변경되었습니다(`fmt: (v) => v.toFixed(1)`).

- 미니노트:
  - "측정된 금리 민감도와 여정이탈률 변화를 기반으로 신규 유입 변화를 추정합니다 (즉시효과)"
  - "여정이탈률 = 대출 여정 중 한도·금리를 확인한 고객이 최종 실행에 도달하지 못한 비율"

#### (5) "추정 손익" 섹션 — 균형추정(월간 기준) (`S.subSectionLabel`)
- 부제: "(균형추정 — 월간 기준)"
- 카드 3종 (`profitTotals`, KpiCard로 렌더), 모두 `toMonthlyMM()`으로 연간(억)→월간(백만원) 환산:

| 라벨 | 단위 | 산출 |
|---|---|---|
| 이자수익 (월) | 백만원 | `toMonthlyMM(sim.tot.intOld) → toMonthlyMM(sim.tot.intNew)` (정수, 콤마) |
| 공헌이익 (월) | 백만원 | `toMonthlyMM(sim.tot.cmOld) → toMonthlyMM(sim.tot.cmNew)` (정수, 콤마) |

## 4. 계산 로직 정의

### 4.1 공통 유틸리티
| 함수 | 정의 |
|---|---|
| `avg(rows, key)` | `rows`의 `key` 값 평균 (빈 배열이면 0) |
| `sumBy(rows, key)` | `rows`의 `key` 값 합계 |
| `toMonthlyMM(v)` | `v × 100 / 12` — 연간(억원) → 월간(백만원) 환산 |

### 4.2 `computeRows(adjMap, base, cofVal)` — 등급별 시뮬레이션
입력: 등급별 조정값 맵(`adjMap`, bp 단위), 기준값 배열(`base` = simBase), 자금원가율(`cofVal`, %)

```
churnNew    = max(0, churn + (bp/10) × elasticity)          // 조정 후 이탈률(%)
retainRatio = (100 − churnNew) / (100 − churn)               // 잔존율
balNew      = balance × retainRatio                          // 균형 상태 잔액(억)

marginNew   = margin + bp/100                                // 조정 후 적용금리(%)
intOld      = balance × margin   / 100                       // 연간 이자수익(억) — 현재
intNew      = balNew  × marginNew / 100                      // 연간 이자수익(억) — 조정 후

cmRateOld   = margin    − cof − legalCostRate − contractCreditRate
cmRateNew   = marginNew − cof − legalCostRate − contractCreditRate
cmOld       = balance × cmRateOld / 100                      // 연간 공헌이익(억) — 현재
cmNew       = balNew  × cmRateNew / 100                       // 연간 공헌이익(억) — 조정 후

opAdjRate   = contractOpRate + creditGapRate + opGapRate − nonIntRate
opOld       = cmOld − balance × opAdjRate / 100              // 연간 영업이익(억) — 현재
opNew       = cmNew − balNew  × opAdjRate / 100              // 연간 영업이익(억) — 조정 후

cntNew      = count  × retainRatio
newWkNew    = newWk  × retainRatio
netWkNew    = netWk  × retainRatio
```
등급별로 위 값을 모두 산출하여 `rows` 배열 반환.

### 4.3 `runSimulation(adjMap, base, cofVal, indirectParams)`
- `computeRows()` 결과(`rows`)를 전체 등급에 대해 `sumBy()`로 합산 → `tot` 객체 (balOld/New, cntOld/New, newWkOld/New, netWkOld/New, intOld/New, cmOld/New, opOld/New)
- `tot.roaOld/roaNew = (opOld/New / balOld/New) × 100` (현재 화면 미표시, 데이터 레이어에는 유지)
- `custGain = tot.cntNew − tot.cntOld` — 신규고객 증가분
- 간접효과:
  - `cardEffect = custGain × (cardRate / 100)`
  - `depositEffect = custGain × depositRate / 10000` (억 단위 — 화면 표시 시 `×100`하여 백만원으로 변환, [[2.4]] 참고)
  - `mauEffect = custGain × (mauRate / 100)`
- 반환: `{ rows, tot, indirect: { custGain, cardEffect, depositEffect, mauEffect } }`

### 4.4 `findOptimalAdj(base, cofVal, { churnCap=15, range=50, step=5 })`
- 각 등급에 대해 `-range`~`+range`bp를 `step`bp 간격으로 전체 탐색(grid search)
- 각 bp 후보에 대해 `churnNew − churn ≤ churnCap`(이탈률 증가폭 15%p 이하)을 만족하는 후보 중,
  `cm = balNew × (marginNew − cof − legalCostRate − contractCreditRate) / 100` (조정 후 연간 공헌이익)이 **최대**가 되는 bp를 선택
- 등급별 최적 bp 맵을 반환 → "최적금리 찾기(참고)" 버튼이 슬라이더에 일괄 적용

### 4.5 ResultPage 집계 로직
- **주간 KPI**: `weekly.filter(phase==='before'/'after')` → 각각 `avg()` → delta% = `(after−before)/before × 100`
- **월간 KPI(고/중저)**: `monthlyHigh/Low.filter(month ∈ beforeMonths/afterMonths)` → 동일 평균/delta 산식
- **통합 월간 차트**: 월별로 `(고신용.metric + 중저신용.metric) × 100` (백만원 환산), `interest=0`인 달(데이터 없음)은 차트에서 제외

> 전체 9개 템플릿의 컬럼별 정의·단위·산식·데이터 추출 가이드는 [`CSV_데이터_가이드.md`](./CSV_데이터_가이드.md)에 상세 기술되어 있으므로 본 절에서는 **업로드 위치/대상/파서**만 요약합니다.

| CSV 파일 | 업로드 위치 | 적용 범위 | 파서 함수 |
|---|---|---|---|
| `template_2_simbase.csv` | 시뮬레이터 탭 상단 "기준값 CSV" | 전체 등급 시뮬레이션 기준값 (전역) | `parseSimBase` |

| `template_3_monthly_low.csv` | 헤더 우측 "중저신용 수익성 CSV (월간)" | 전체 이벤트 공유 (중저신용 1~6월) | `parseMonthly` |

- 공통 파서 `parseRows(text, {stringCols})`: 첫 줄을 헤더로 사용, 콤마 구분, `stringCols`에 명시된 컬럼(`grade`,`week`,`phase`,`month`)을 제외한 모든 값은 숫자로 변환
- `readCsvFile(file, parser, onParsed, errMsg)`: `FileReader`로 텍스트 읽기 → 파싱 실패 시 `errMsg`로 `alert()` 표시
- **컬럼명은 코드의 필드명과 정확히 일치해야 함** (대소문자 포함). 일치하지 않으면 해당 값이 `undefined`로 처리되어 계산 결과가 `NaN`이 될 수 있음 (이전 점검에서 `template_2_simbase.csv`의 `interestRate`→`margin` 불일치를 발견하여 수정 완료)
- **v0.2 데이터 정밀도**: `template_2_simbase.csv`의 `balance`/`newWk`/`netWk`, `template_1_weekly_ev*.csv`의 `newLoan`/`netLoan`/`balance`는 화면에 소수점 1자리까지 표시되므로, CSV 값도 정수가 아닌 소수 1자리 단위로 입력하는 것을 권장 (정수 입력 시에도 동작에는 문제없음). 데모 템플릿도 소수 1자리 값으로 갱신됨.

## 7. 미결정 / 추후 협의 항목 (Open Items)

1. **ADW 데이터 연동**: 현재 수익성 지표(이자수익/공헌이익/영업이익)는 CSV 업로드 또는 데모 데이터 기반. ADW 데이터 구축 완료 후 전략등급별 실측치로 자동 연동 예정 (현재는 수동 CSV 교체 방식).
2. **탄력성 계수(`elasticity`) 및 원가율 6종**: 현재 가정값(`DEMO_SIM_BASE`)이며, 과거 금리 변경 이벤트의 실제 이탈률 변화 데이터를 확보하면 역산하여 교체 필요 ([`CSV_데이터_가이드.md`](./CSV_데이터_가이드.md) 1번 항목 참고).
3. **자금원가율(cof)**: 현재 화면에서 직접 입력(기본값 1.7%)하는 방식. 실시간 연동 데이터 소스가 확정되면 자동 반영 방식으로 전환 검토.
4. **간접효과 전환율(카드발급/예금평잔/MAU, n%)**: 현재 사용자가 화면에서 직접 입력하는 가정값(기본 35%/120만원/60%). 실측 전환율 확정 시 고정값 또는 별도 CSV로 대체 검토.
5. **시뮬레이터 서브섹션 캡션 검증**: "신규유입 변화 (즉시효과 — 금리 조정 시 단기 반응)" / "추정 손익 (균형추정 — 월간 기준)" 캡션은 이전 복구 작업 중 재구성된 문구로, 원래 의도와 정확히 일치하는지 최종 확인 필요 (관련: 내부 메모 `feedback-corruption-recovery`).
6. **고신용/중저신용 분류 기준의 실데이터 매핑**: 현재 "신용평가 기준"이라는 설명만 존재. 실제 신용평가 등급 컷오프(예: 몇 등급 이상을 고신용으로 분류하는지) 문서화 필요.
7. **이벤트(EVENTS) 확장성**: 현재 3개 이벤트(`ev331`,`ev423`,`ev529`)가 코드에 하드코딩됨. 향후 신규 금리 변경 이벤트 추가 시 `EVENTS` 배열 + 대응 CSV 3종(주간/퍼널) + 월간 CSV의 `beforeMonths`/`afterMonths` 구간을 함께 갱신해야 함.
8. **백업/버전 관리**: `standalone(backup0612).html`은 2026-06-12 시점 스냅샷으로, 이후(로고/eyebrow/막대 크기/LabelList/소수점 표시/단위 변경 등) 변경이 반영되지 않은 상태. 최신 스냅샷 백업 필요 여부 확인.
9. **연동 예금평잔(백만원) 표시값의 현실성 검토**: 단위를 억→백만원으로 변경하면서 숫자가 100배 커짐(예: 1.2억 → 120백만원). 자릿수가 늘어난 표시가 보고용으로 적절한지(혹은 천단위 콤마만으로 충분한지) 실데이터 입력 후 재검토 권장.
